## 1. 现状分析

  - `POST /resumes/{id}/optimize-pipeline` — 6阶段Pipeline（能力评估→人岗匹配→故事生成→简历组装→ATS优化→人工把关）

### 1.2 痛点识别

| 问题 | 影响 |
|------|------|
| 复用 Pinme 共享 Key，多人共用配额 | 频率受限，高峰期可能排队或拒绝 |
| gpt-4o-mini 处理复杂简历分析能力有限 | 故事生成质量低、关键词提取不精准 |

| 无多模型切换能力 | 无法按任务复杂度路由不同模型 |

## 2. 简历优化5个分析维度

### 维度1：内容优化建议

**目标**：提升简历文案质量，使每条描述都具有说服力。

| 子维度 | 分析重点 | AI 介入点 |
|--------|---------|----------|
| 文案表达 | 主动语态 vs 被动语态、动词选择力度、句式多样性 | 逐句建议替换，输出 before/after 对比 |
| 亮点提炼 | 经历中最具差异化的成就提取 | 从大段描述中抽取 1-2 个核心亮点，前置到 summary |
| 成果量化 | 检查是否存在未量化的成果描述 | 标注缺少数字的成就句，提示可量化维度（金额、效率、人数、规模） |
| 冗余删减 | 识别与目标岗位无关的内容 | 建议删除/降级不相关内容 |

**AI Prompt 策略**：提供简历全文 + JD，要求逐条分析并给出具体修改文案，而非笼统建议。

### 维度2：关键词匹配分析

**目标**：使简历在 ATS 和 HR 人工筛选中获得更高相关性评分。

| 子维度 | 分析重点 | AI 介入点 |
|--------|---------|----------|
| ATS 关键词覆盖 | JD 中的硬技能词（工具、技术、认证） | 提取 JD 关键词 → 检索简历 → 输出缺失词和注入位置 |
| 行业术语匹配 | 行业黑话、缩写、方法论名称的准确使用 | 检查术语是否使用了行业内通用写法 |
| 关键词密度 | 核心关键词出现频率 | 统计密度，提示过少（<1次）或堆砌（>5次） |
| 语义等价 | 同义词/近义词替换 | 识别简历用词与 JD 用词的差异，建议对齐 |

**输出示例**：
```
JD 关键词"敏捷开发"未出现在简历中 → 建议在项目描述中补充"采用 Scrum 双周迭代"
```

### 维度3：格式建议

**目标**：确保简历在视觉和信息层级上专业、易读。

| 子维度 | 分析重点 | AI 介入点 |
|--------|---------|----------|
| 排版规范 | 字体/字号一致性、页边距、行间距 | 规则检查（不需要 AI） |
| 段落结构 | 每段行数、bullet 数量、信息密度 | AI 分析段落是否过长，建议拆分或合并 |
| 视觉层次 | 标题、副标题、正文的层级关系 | AI 评估信息组织逻辑是否清晰 |
| 篇幅控制 | 总页数（1.5-2页为佳）、各模块占比 | 统计字数，给出各模块占比建议 |

### 维度4：ATS 兼容性

**目标**：确保简历能被主流的 Applicant Tracking System 正确解析。

| 子维度 | 分析重点 | AI 介入点 |
|--------|---------|----------|
| 机器解析友好度 | 是否使用了表格/图片/特殊符号 | 规则检测 + AI 提示替代方案 |
| 标准字段映射 | 姓名、联系方式、工作经历、教育背景是否明确标识 | AI 识别缺失字段并提示 |
| 文件格式 | PDF vs Word 对不同 ATS 的兼容性 | 规则建议（投递时 PDF + Word 备选） |
| 特殊字符/编码 | 是否存在导致解析乱码的字符 | 规则检查 |

### 维度5：人岗匹配度

**目标**：量化评估候选人与目标岗位的匹配程度，提供差距分析和弥补策略。

| 子维度 | 分析重点 | AI 介入点 |
|--------|---------|----------|
| 硬技能匹配 | 学历、年限、技术栈、行业经验 | 逐项比对，输出匹配矩阵（✅/⚠️/❌） |
| 软技能匹配 | 沟通、管理、学习能力 | AI 从简历描述中推断软技能，与 JD 对比 |
| 综合评分 | 0-100 分的整体匹配度 | 加权计算，含置信度说明 |
| 差距弥补 | 弱项的具体提升建议 | AI 生成可操作的弥补策略（证书、项目、学习路径） |

## 3. API 选型对比

### 3.1 方案概览

| 对比项 | 方案A：Pinme共享API | 方案B：独立API |
|--------|-------------------|---------------|
| 模型 | gpt-4o-mini | gpt-4o / gpt-4o-mini（可选） |
| 接入方式 | Pinme 代理转发，共享 API Key | 直连 OpenAI/Azure，独立 Key |
| 成本归属 | 与其他项目共用配额 | 独立计费，完全自主 |
| 并发能力 | 共享池，高峰期可能排队 | 独立配额，按 tier 分级 |
| 上下文窗口 | 128K tokens | 128K tokens（gpt-4o） |
| 适用场景 | MVP 验证、低频使用 | 生产环境、高频调用、长文档分析 |

### 3.2 模型能力对比

| 维度 | gpt-4o-mini（方案A） | gpt-4o（方案B） |
|------|---------------------|----------------|
| 推理能力 | 中 | 高（复杂匹配分析更准确） |
| 中文质量 | 良好 | 优秀（故事生成更地道） |
| JSON 格式遵循 | 一般（偶有格式错误） | 优秀 |
| 长文档理解 | 基本可用 | 优秀（JD+简历全文一次处理） |
| 响应速度 | 快（~2-3秒） | 较快（~4-6秒） |
| 上下文窗口 | 128K | 128K |

### 3.3 成本对比

**单次调用假设**：
- 6阶段Pipeline全流程：System Prompt（~1000 tokens）+ 简历（~1000 tokens）+ JD（~1000 tokens）= ~3000 tokens input
- 输出6个阶段完整内容：~2000 tokens output
- 合计：~5000 tokens/次

| 计费项 | gpt-4o-mini | gpt-4o |
|--------|------------|--------|
| Input 单价 | $0.15 / 1M tokens | $2.50 / 1M tokens |
| Output 单价 | $0.60 / 1M tokens | $10.00 / 1M tokens |
| 单次成本（5000t） | $0.00165 ≈ ¥0.012 | $0.0275 ≈ ¥0.20 |

### 3.4 月度成本估算

**使用频次假设**：
- 简历优化（Pipeline）：4次/月
- 简历能力评估（单次 optimize）：4次/月
- 合计：8次/月

| 方案 | 单次成本 | 月成本（8次） | 年成本 |
|------|---------|-------------|--------|
| 方案A（gpt-4o-mini） | ¥0.012 | ¥0.10 | ¥1.15 |
| 方案B（gpt-4o） | ¥0.20 | ¥1.60 | ¥19.20 |

> **结论**：两种方案成本都极低。方案A几乎免费（但依赖Pinme共享配额），方案B年成本也仅约 ¥20。

### 3.5 推荐方案

**推荐：方案B（独立API + gpt-4o），理由如下**：

1. 成本可忽略（¥20/年），不是决策因素
2. gpt-4o 在复杂简历分析和中文故事生成上显著优于 gpt-4o-mini
3. 独立 Key 不受其他项目配额影响，不会出现"高峰期不可用"
4. JSON 格式遵循更好，减少解析失败导致的降级
5. 后续可以灵活切换 gpt-4o / gpt-4o-mini 做成本优先级的任务分流

**实施建议**：
- 复杂 Pipeline（6阶段）：路由到 gpt-4o
- 简单优化（单次 optimize）：路由到 gpt-4o-mini（降成本）
- 后续接入 Claude Sonnet 4 做备选模型（Claude 在中文简历故事生成上可能更好）

## 4. 降级策略设计

### 4.1 降级触发条件

| 优先级 | 场景 | 触发条件 |
|--------|------|---------|
| 1 | AI 服务不可用 | HTTP 非 2xx、超时(>30s)、连接失败 |
| 2 | AI 返回格式错误 | JSON 解析失败 3 次重试后 |
| 3 | Token 配额耗尽 | API 返回 429 / insufficient_quota |
| 4 | 安全过滤 | 内容被标记为不安全 |

### 4.2 降级层级

```
调用 AI API
    ├── 成功 → 返回 AI 结果
    ├── 失败（可重试错误）→ 重试（最多3次，指数退避 1s/2s/4s）
    │     ├── 重试成功 → 返回 AI 结果
    │     └── 全部失败 → 进入降级
    └── 直接失败 → 进入降级

降级层：
    ├── 第1层：规则引擎分析（关键词匹配 + 格式检查 + 模板匹配）
    └── 第2层：返回静态提示（告知用户 AI 不可用，建议稍后重试）
```

### 4.3 规则引擎实现方案

当 AI 不可用时，启动基于规则的简历分析：

```python
# 伪代码结构
class RuleBasedResumeAnalyzer:
    def analyze(self, resume_content: str, jd_text: str) -> dict:
        return {
            "keyword_analysis": self._keyword_match(resume_content, jd_text),
            "format_check": self._format_check(resume_content),
            "match_score": self._calculate_match_score(resume_content, jd_text),
            "basic_suggestions": self._generate_suggestions(resume_content, jd_text),
        }
```

#### 4.3.1 关键词词频统计

- 使用 jieba 分词提取 JD 中的高频名词/动词（频次 > 2 的前 15 个词）
- 在简历中检索每个关键词出现次数
- 输出：覆盖率百分比 + 缺失词列表 + 建议注入位置

#### 4.3.2 格式检查

| 检查项 | 规则 | 严重度 |
|--------|------|--------|
| 页数 | 字数 > 2000 提示可能超 2 页 | 提示 |
| 联系方式 | 未检测到手机号/邮箱 | 严重 |
| 日期格式 | 检查是否存在不一致的日期格式 | 提示 |
| 特殊字符 | 检测表格、图片标记 | 警告 |
| 段落长度 | 单段 > 300 字建议拆分 | 提示 |

#### 4.3.3 标准模板匹配

- 预置 3 套简历模板（技术岗 / 管理岗 / 通用岗）
- 根据 JD 中的岗位类型关键词匹配最佳模板
- 将用户简历内容按模板结构重组，输出建议结构

### 4.4 前端降级提示

```tsx
// AI 不可用时显示
{isFallback && (
  <Alert variant="warning">
    AI 服务暂时不可用，已生成基于规则的分析结果。分析精度有限，
    建议稍后重试获取 AI 深度优化方案。
  </Alert>
)}
```
