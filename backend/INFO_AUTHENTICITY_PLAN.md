# 岗位信息真实性综合解决方案

> 版本：v1.0 · 创建于 2026-05-30
> 目的：解决求职助手Web岗位数据的真实性、完整性和可靠性问题
> 当前状态：单数据源（BOSS直聘 via autocli），jd_url为空率高，无验证机制

---

## 一、信息源评估（量化评分）

### 1.1 BOSS直聘（通过autocli抓取）

| 维度 | 评分 | 说明 |
|------|------|------|
| **信息完整性** | 7/10 | 公司名/薪资/标题/地区数据完整；JD正文通常500-2000字；但jd_url（详情链接）有30%概率为空或加密失效 |
| **信息真实性** | 8/10 | BOSS直聘有企业认证+HR实名+营业执照审核机制，虚假职位率低于其他平台；高端钓鱼岗位（薪资异常高）约占3-5% |
| **链接稳定性** | 4/10 | BOSS直聘详情页URL含一次性securityId，24-48小时内失效；通过页面跳转访问而非直接URL |
| **数据量** | 9/10 | 国内招聘市场覆盖面最广的平台之一，单个关键词可返回200+岗位；杭州地区AI/互联网岗位覆盖充分 |

**综合评分：7.0/10**

当前autocli实现从列表页提取数据，字段包含 name/company/salary/area/url/description/skills/boss/degree/experience，覆盖率约85%。主要损失在：详情页全文JD（列表页只展示截断描述）和稳定可回溯的URL。

**改进方向**：在autocli中增加 `--detail` 模式逐岗位进入详情页提取完整JD；URL稳定性问题无解（平台反爬设计）。

### 1.2 猎聘（当前无数据源）

| 维度 | 评分 | 说明 |
|------|------|------|
| **信息完整性** | 8/10 | 猎聘JD通常比BOSS直聘更详细（中高端岗位），薪资范围更透明；要求猎聘已登录 |
| **信息真实性** | 9/10 | 猎聘平台审核最严格（中高端定位），企业认证+职位审核双机制；虚假岗位率最低 |
| **链接稳定性** | 5/10 | 猎聘详情页URL也含时效token，但相对BOSS直聘稍稳定（一周级别） |
| **数据量** | 6/10 | 猎聘中高端覆盖好但总量不如BOSS直聘；AI/互联网岗位杭州地区约BOSS直聘的60% |

**综合评分：7.0/10**（数据量略低但真实性高）

**接入可行性**：autocli支持 `liepin search` 命令，原理与BOSS直聘相同（Chrome扩展提取DOM）。已验证可返回岗位数据。需Chrome中保持猎聘登录态。

**当前gap**：`try_autocli_search` 函数仅硬编码了 `boss` 命令，未实现猎聘分支。扩展成本低（约20行代码）。

### 1.3 虚假信息识别指标体系（5+指标）

| 指标 | 检测方法 | 权重 | 阈值 |
|------|---------|------|------|
| **JD过于简略** | jd_text字数 < 100 或仅含标签无正文 | 15% | <100字 |
| **公司信息缺失** | company为空或为"某XX公司""XX科技"等泛化名称 | 15% | 模糊匹配 |
| **薪资异常** | salary_min > city_avg × 2 或 salary_min < city_min_wage | 20% | 杭州 >80K或 <3K |
| **入职收费信号** | JD含"培训费""押金""保证金""垫付"等关键词 | 25% | 正则匹配命中 |
| **伪造利好信号** | JD含"不要求经验""日结""在家可做""轻松月入过万"且薪资异常 | 15% | 组合规则 |
| **公司存续状态** | 国家企业信用信息公示系统查询：注销/吊销/列入异常 | 10% | API查询结果 |

**公式**：`风险分数 = Σ(指标命中 × 指标权重)`

```
风险分数 ≥ 0.5 → 高危（标注红色警告，建议用户核实）
风险分数 0.3-0.5 → 可疑（标注黄色警告）
风险分数 < 0.3 → 正常
```

## 三、账号联动方案设计

### 3.1 Cookie管理方案

**当前状态**：autocli通过Chrome扩展连接已登录的Chrome浏览器读取页面数据，Cookie由Chrome浏览器管理。

**获取方案**：

| 方案 | 方式 | 优缺点 |
|------|------|--------|
| **方案A：autocli扩展（当前）** | Chrome扩展注入已登录页面 → 无需手动处理Cookie | ✅零配置 ⚠️必须保持Chrome打开+BOSS已登录 |
| **方案B：pyCookie导出** | Chrome插件导出Cookie JSON → Python写入requests会话 | ✅可离线运行 ⚠️需手动导出，易过期 |
| **方案C：CDP Network.getCookies** | 通过CDP协议从浏览器获取Cookie → 持久化到本地文件 | ✅可自动化刷新 ⚠️需CDP服务常驻 |

**存储方案**：

```
后端存储：SQLite cookies 表（加密存储）
字段：platform / cookie_name / cookie_value(encrypted) / domain / expires / created_at
加密方式：Fernet对称加密（cryptography库，密钥存环境变量）
```

**刷新策略**：
- 每次搜索前检查Cookie是否在有效期内（距过期 > 24小时）
- 已过期 → 提示用户重新登录（前端弹窗引导）
- 用户重新登录后 → CDP自动获取新Cookie → 更新数据库

### 3.2 Session维持策略

```
┌──────────────────────────────────────────────────────┐
│                 Session 生命周期管理                     │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌─────────┐    ┌─────────┐    ┌──────────┐        │
│  │ 活跃态   │───▶│ 休眠态   │───▶│ 过期态    │        │
│  │ (0-24h) │    │ (1-7天) │    │ (>7天)   │        │
│  └─────────┘    └─────────┘    └──────────┘        │
│       │              │              │               │
│       ▼              ▼              ▼               │
│  正常抓取       轻声心跳检查      提示重新登录           │
│                  每4h一次                            │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**策略细则**：

| 策略 | 实现 |
|------|------|
| **定期活跃** | 后端定时任务（每6小时），使用CDP访问BOSS直聘首页保持活跃度 |
| **心跳检测** | 请求BOSS直聘用户信息API（`/wapi/zpuser/info`），检查返回码 |
| **异常检测** | 响应403（登录过期）/302（重定向到登录页）/验证码页面 → 标记Cookie无效 |
| **重新登录** | 前端弹窗 → 用户手动完成登录 → CDP抓取新Cookie → 更新数据库 |
| **降级策略** | Cookie失效期间，搜索降级为Mock数据 + 前端明确提示"需要重新登录获取实时岗位" |

### 3.3 API授权

**招聘平台开放API现状**：

| 平台 | 开放API | 申请条件 | 数据范围 | 费用 |
|------|---------|---------|---------|------|
| BOSS直聘 | 企业版开放平台（bosszp.com） | 需企业营业执照+HR实名认证 | 发布职位/收简历/沟通 | 企业套餐（年费万级） |
| 猎聘 | 猎聘开放平台（open.liepin.com） | 需企业认证 | 职位发布/简历搜索 | 企业套餐 |
| 拉勾 | 拉勾开放平台（open.lagou.com） | 需企业认证 | 职位管理/简历管理 | 企业套餐 |
| 智联招聘 | 智联卓聘API | 需商业合同 | 职位+简历 | 商务定价 |
| 前程无忧 | 51job企业API | 需商业合同 | 职位发布+简历搜索 | 商务定价 |

**结论**：当前阶段（个人求职场景）不适合走官方API路线（需要企业资质+商业付费）。优先保持autocli/CDP抓取方案。

**例外**：如有朋友的企业账号，可通过企业版API获取结构化数据（优于抓取）。执行时需确认数据使用范围不违反平台服务条款。

### 3.4 当前autocli方案的优势与局限

| | 优势 | 局限 |
|------|------|------|
| **架构** | CLI轻量、零配置部署；Chrome扩展提取真实页面数据 | 依赖Chrome打开+已登录；Mac版稳定性待验证 |
| **数据质量** | 直接读取DOM，数据准确；支持55+平台 | 详情页深度提取不稳定（URL失效）；仅列表页数据 |
| **可维护性** | autocli由开源社区维护（nashsu/opencli-rs） | 平台改版后需等待autocli更新；跨平台升级依赖第三方 |
| **扩展性** | 已支持BOSS/猎聘/51job/拉勾等多平台 | 非平台命令适配需单独开发；无法自定义提取规则 |
| **稳定性** | 已验证可返回真实岗位数据 | subprocess调用有超时风险（单次搜索13-20s）；city参数偶发不生效 |
| **合规性** | 使用用户自己的浏览器+登录态 | 法律边界灰色（非商业用途个人使用风险可控） |

**建议**：P1阶段保持autocli作为主数据源，同时开发备用的Python直连抓取方案（requests + Cookie导入）作为降级方案。

## 四、信息交叉验证机制

### 4.1 企业信用验证流程

```
┌─────────────────────────────────────────────────────────┐
│            国家企业信用信息公示系统验证流程                  │
│            (http://www.gsxt.gov.cn)                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  输入：公司全称                                          │
│       │                                                 │
│       ▼                                                 │
│  1. 查询企业基本信息                                      │
│     - 统一社会信用代码                                    │
│     - 法定代表人                                         │
│     - 注册资本                                           │
│     - 成立日期                                           │
│     - 经营范围                                           │
│       │                                                 │
│       ▼                                                 │
│  2. 检查企业状态                                          │
│     - 存续/在业 → 正常                                   │
│     - 注销/吊销 → 高危（红色警告）                        │
│     - 列入经营异常 → 可疑（黄色警告）                      │
│     - 列入严重违法失信 → 高危（红色警告）                  │
│       │                                                 │
│       ▼                                                 │
│  3. 输出验证报告                                          │
│     - 企业状态 + 风险等级                                │
│     - 成立年限（新公司 < 1年 → 标记）                    │
│     - 注册资本（< 100万 → 标记）                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**技术实现**：
- gsxt.gov.cn 无反爬但有验证码（OCR或手动打码）
- **替代方案**：使用天眼查/企查查API（有稳定接口，成本低）

### 4.2 公司官网验证

```
验证流程：
1. 从JD/公司名推断官网域名 → 搜索引擎确认
2. 官网HTTP状态检查（200/404/301）
3. 官网内容扫描：
   - 是否有招聘页面匹配该岗位？
   - 公司介绍是否与JD中的公司描述一致？
   - 新闻/动态时间线（是否在正常运营？）
4. 官网权重评分（有无HTTPS/ICP备案/社交媒体链接）
```

**验证规则**：
- 官网404/域名过期 → 橙色警告
- 官网无招聘页面 → 可能是代理招聘或外包 → 黄色标记
- 官网信息与JD描述严重不符 → 红色警告

### 4.3 天眼查/企查查API接入可行性

| 维度 | 天眼查 | 企查查 |
|------|--------|--------|
| **公开API** | 有（开放平台open.tianyancha.com） | 有（开放平台openapi.qcc.com） |
| **免费额度** | 个人开发者100次/天 | 个人开发者50次/天 |
| **付费套餐** | 0.1元/次起 | 0.05元/次起 |
| **数据覆盖** | 2.8亿+企业 | 2亿+企业 |
| **核心字段** | 企业基本信息/股东/高管/风险信息 | 企业基本信息/司法风险/经营风险 |
| **接入难度** | ★★☆☆☆（RESTful API，Python SDK） | ★★☆☆☆（RESTful API，Python SDK） |
| **月度成本估算** | 30天×20次/天×0.1元 = 60元/月 | 30天×20次/天×0.05元 = 30元/月 |

**推荐方案**：企查查API（单价更低，数据质量相当）+ 国家企业信用信息公示系统免费查询（兜底）。

### 4.4 多源数据对比验证规则

```
同一公司+同一岗位在多平台（BOSS直聘/猎聘/拉勾）的数据对比：

┌──────────────┬──────────────────────────────────────────┐
│ 对比项        │ 验证规则                                  │
├──────────────┼──────────────────────────────────────────┤
│ 公司名        │ 三平台一致（允许包含/简称差异，模糊匹配90%）  │
│ 薪资范围      │ 偏差 < 30%（超过 → 标记薪资不确定）         │
│ JD关键职责    │ 核心关键词重叠度 > 60%（低于 → 可能为虚假JD）│
│ 岗位发布时间   │ 偏差 < 7天（超过 → 可能为过期岗位）         │
│ 公司规模      │ 标注是否一致（小型/中型/大型）               │
│ 工作地点      │ 偏差 < 5km（城市内）                       │
└──────────────┴──────────────────────────────────────────┘

一致性评分 = 一致项数 / 总对比项数
≥ 0.8 → 高可信度
0.5-0.8 → 中等可信度（存在疑问，建议核实）
< 0.5 → 低可信度（红色警告，不建议投递）
```
